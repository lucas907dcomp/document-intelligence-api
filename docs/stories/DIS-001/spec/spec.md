## 1. Overview

The **Document Intelligence Service** is a backend RAG (Retrieval-Augmented Generation) API built with Java 21 and Spring Boot. It receives PDF uploads via REST, stores them temporarily on a shared Docker volume using the Claim Check Pattern, processes them asynchronously via Apache Kafka (extracting text, chunking, and generating embeddings via OpenAI through Spring AI), and persists the resulting vectors in PostgreSQL with pgvector. A Q&A endpoint accepts natural language questions, performs semantic similarity search over the stored embeddings, and returns contextualized answers generated by GPT-4o-mini.

*Derived from: FR-1, FR-2, FR-3, FR-4, CON-1, CON-2, CON-3, ADR-001*

### 1.1 Goals

- Enable asynchronous PDF ingestion via Event-Driven Architecture (Kafka) — *FR-1, FR-2, CON-2*
- Store and query document embeddings via pgvector similarity search — *FR-2, FR-4, CON-3*
- Expose a natural language Q&A endpoint grounded in document context — *FR-4*
- Abstract LLM and embedding providers via Strategy Pattern (Spring AI) — *FR-5*
- Provide complete OpenAPI 3 documentation via Swagger UI — *FR-6*
- Route failed messages to a Dead Letter Queue for operational resilience — *FR-7*
- Run entirely via `docker-compose up` — *NFR-4, CON-2, CON-3*

### 1.2 Non-Goals

- Authentication or authorization (explicitly out of scope for MVP) — *CON-5*
- OCR for scanned or image-based PDFs — *CON-6*
- Conversation history / multi-turn chat — *ASM-2*
- Multi-tenant document isolation — *ASM-3*
- Horizontal consumer scaling across multiple Docker hosts (single-host MVP) — *CON-7*
- Frontend or UI layer

## 3. Technical Approach

### 3.1 Architecture Overview

Single Spring Boot application hosting both the REST API and the Kafka consumer (single deployable unit for MVP). All components communicate via defined interfaces to support the Strategy Pattern.

```
Client
  │
  ▼
[REST API — Spring Boot]
  │  POST /documents
  │  ├─ Validate (type, size)
  │  ├─ Write PDF → Shared Volume (CON-7, ADR-001)
  │  ├─ Create Document{status=PROCESSING} → PostgreSQL
  │  └─ Publish DocumentUploadedEvent{documentId, filePath} → Kafka
  │
  │  GET /documents/{id}/status → PostgreSQL
  │  POST /documents/{id}/query → pgvector → OpenAI → response
  │
[Kafka Broker]
  │
  ▼
[Kafka Consumer — same Spring Boot app]
  ├─ Read filePath from event
  ├─ Open PDF from Shared Volume
  ├─ Extract text (PDFBox)
  ├─ Chunk text (1000 chars / 200 overlap)
  ├─ Generate embeddings (EmbeddingProvider → OpenAI)
  ├─ Persist Chunk + Embedding → PostgreSQL/pgvector
  ├─ Update Document{status=READY}
  └─ Delete PDF from Shared Volume
  │
  └─ On failure → DLQ topic + Document{status=FAILED}

[PostgreSQL + pgvector]
  Tables: documents, chunks, embeddings

[Shared Docker Named Volume]
  Path: {PDF_STORAGE_PATH}/{documentId}/{filename}
```

*Derived from: FR-1, FR-2, FR-3, FR-4, CON-2, CON-3, CON-7, ADR-001, INT-1, INT-2, INT-3, INT-4*

### 3.3 Data Flow

#### Upload Flow *(INT-1)*

```
POST /documents (multipart/form-data)
  → Validate: content-type=application/pdf, size ≤ 10MB    [CON-4]
  → StorageService.store(documentId, file)                  [CON-7, ADR-001]
      writes to: {PDF_STORAGE_PATH}/{documentId}/{filename}
  → DocumentEventProducer.publish(DocumentUploadedEvent{documentId, filePath})
      [on KafkaException]: StorageService.delete(filePath) → return 503, NO DB write  [EC-1]
  → Document{id, filename, storagePath, status=PROCESSING, uploadedAt, fileSizeBytes}
      persisted to PostgreSQL  ← only after successful Kafka publish
  → Return: 202 Accepted { documentId }
```

#### Processing Flow *(INT-4)*

```
@KafkaListener receives DocumentUploadedEvent{documentId, filePath}
  → StorageService.retrieve(filePath)           read PDF bytes from volume
  → PdfExtractor.extract(bytes)                 plain text
  → TextChunker.chunk(text)                     List<Chunk> (1000/200)
  → for each chunk:
      EmbeddingProvider.embed(chunk.content)    float[] vector (1536 dims)
      persist Chunk + Embedding (single tx)
  → Document.status = READY
  → StorageService.delete(filePath)             claim check cleanup
  → On failure:
      Document.status = FAILED
      publish to DLQ topic
      log structured error (documentId, filePath, exception)
      skip file deletion (preserve for inspection)
```

#### Query Flow *(INT-3)*

```
POST /documents/{id}/query { question }
  → Verify Document exists + status=READY
  → EmbeddingProvider.embed(question)           question vector
  → EmbeddingRepository.findTopK(docId, vector, K=5)
      pgvector cosine similarity search          [OQ-3 resolved: K=5]
  → Prompt assembly:
      system: "Answer based only on context below"
      context: chunk contents (top-5)
      user: question
  → LlmProvider.complete(prompt)               GPT-4o-mini
  → Return: { answer, sourceChunks }
```

## 7. Risks & Mitigations

| Risk | Probability | Impact | Mitigation |
|------|------------|--------|-----------|
| OpenAI API rate limits during bulk embedding (many chunks per doc) | Medium | High | Implement exponential backoff in `OpenAiEmbeddingProvider`; configurable retry via Spring Retry |
| pgvector cosine search latency degrades with high embedding count | Low (MVP scale) | Medium | Add IVFFlat index on embedding vector column: `CREATE INDEX ON embeddings USING ivfflat (vector vector_cosine_ops)` |
| PDF text extraction failure (corrupted, encrypted, non-text) | Low–Medium | Medium | Caught in consumer; Document→FAILED + DLQ; EC-2 covered in acceptance tests |
| Shared volume disk exhaustion under consumer lag | Low | High | Mitigated by 10MB file size limit (CON-4) + immediate cleanup post-processing (INT-4) |
| File missing from volume when consumer processes event (EC-7) | Very Low | High | Handled explicitly: FAILED status + DLQ; covered in acceptance tests |
| Kafka consumer offset not committed on failure (double-processing) | Low | Medium | Use `MANUAL_IMMEDIATE` ack mode; commit only after successful DB write + file deletion |
